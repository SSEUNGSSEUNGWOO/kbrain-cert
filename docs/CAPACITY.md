# kbrain-cert — 300명 동시 응시 실행 계획 (Capacity Plan)

**목표**
- 응시자 300명이 동시 응시해도 **답안 유실 0건**
- **응답 p95 < 500ms** (문제 페치 · 답안 저장 · 서버 시간 동기)
- 감독 이벤트 유실률 < 0.1%
- 녹화 청크 유실률 < 1% (재시도 포함)
- 시험 종료 순간 최종 제출 폭주(300명이 동시 클릭) 안전 처리

**미결정 5개는 하단 §7 참고** (계정 발급 · 계약 필요 항목).

## 3. 병목별 상세 대응 (각 병목마다 · 원인 · 대응 · 검증 · 잔여 리스크)

### 3.1 시험 시작 순간 문제 페치 폭주

- **원인**: 300명이 정각에 응시 시작 → 문제 payload 300 요청 폭주 (동일 payload)
- **대응**:
  - Vercel Edge Cache로 문제 payload 캐싱 (`unstable_cache` + tag revalidation)
  - Supabase는 서버 컴포넌트에서 뷰 조회 → 캐시 히트 시 DB 요청 없음
  - 응시자별 시작 시각 랜덤 지연 ±3s 옵션 (thundering herd 완화)
- **검증**: Playwright + Artillery로 초당 300 페치 시뮬레이션 · p95 < 400ms 목표
- **잔여 리스크**: 캐시 무효화 타이밍 (관리자가 시험 시작 직전 문제 수정)

### 3.2 답안 저장 지속 write

- **원인**: 300명 × 문제당 debounce 3s → 지속적 upsert 트래픽
- **대응**:
  - PgBouncer transaction mode 필수 (물리 커넥션 압축)
  - 응답 IndexedDB 로컬 백업 → 네트워크 순단 시 대기 후 flush
  - Server Action idempotent (session_id + question_id 유니크 upsert)
- **검증**: 부하 스크립트로 300 세션 × 5분간 지속 저장 · 유실 0 확인
- **잔여 리스크**: 브라우저 오프라인 시간이 길면 IndexedDB 데이터 손실 (사용자 브라우저 정리 등)

### 3.3 감독 이벤트 폭주

- **원인**: 300명 × 초당 2 이벤트 = **600 writes/s peak**
- **대응**:
  - **클라에서 5s 윈도우 배치** → 초당 60~120 bulk insert만
  - 이벤트 테이블 파티셔닝 (month + session_id)
  - Realtime inbound는 관리자만 (그리드 진입 시)
- **검증**: 300 명 × 5분 시뮬 · 이벤트 유실률 < 0.1%
- **잔여 리스크**: 배치 대기 중 브라우저 크래시 → 최근 5s 이벤트 손실 (수용 가능)

### 3.4 Daily.co 300 SFU 스트림

- **원인**: 감독관 그리드에서 30~50 명 receive 시 대역폭 폭증
- **대응**:
  - 감독관 그리드는 **저화질 썸네일** 요청 (Daily simulcast)
  - 이벤트 발생 응시자만 고화질 확대 (3단 정렬 UI에서 자동)
  - 감독관 5명 이내 · 각자 60명 담당 (실무 리허설에서 조정)
- **검증**: 실기기 리허설 필수 (Playwright는 WebRTC 미지원)
- **잔여 리스크**: 감독관 노트북 사양 · 회사 네트워크 대역폭

### 3.5 R2 청크 업로드 병목

- **원인**: 300 명 × 2 트랙 × 초당 2 chunks = **1,200 writes/s peak**
- **대응**:
  - Presigned URL로 클라 → R2 직접 (서버 개입 없음)
  - **청크 5초 단위로 늘려 write 수 1/10** (비용·병목 동시 완화)
  - 실패 시 IndexedDB 큐 + 3회 재시도
- **검증**: 실기기 30~60명 리허설로 실측
- **잔여 리스크**: 응시자 네트워크 순단 시 청크 누락 (수용 가능 · 사후 검토용)

### 3.6 최종 제출 순간 폭주

- **원인**: 시험 종료 시각 근처 300 명이 동시에 최종 제출 클릭
- **대응**:
  - Server Action idempotency key (`session_id`) 강제
  - 자동 제출은 클라에서 순차 발동(응시자마다 미세 시차)
  - 서버는 락 없이 upsert (한 번만 처리되도록 응답 상태 확인)
- **검증**: 300 세션 종료 시각을 초 단위로 동시 발동 시뮬
- **잔여 리스크**: DB 커넥션 풀 소진 시 일부 요청 지연 → idempotent라 재시도 안전

### 3.7 브라우저 부하 (WebRTC + MediaRecorder + face-api + WebAudio 동시)

- **원인**: 응시자 노트북에서 4개 스트림 처리 동시
- **대응**:
  - 대기실에서 CPU 벤치마크 측정 (경쟁하는 4개 작업 시뮬)
  - 최소 사양 명시 (Intel i5 8세대 or Apple M1 이상 권장)
  - 저사양 응시자는 관리자 예외 승인 · 신분증 인증 강화
  - face-api 감지 주기 2.5s → 저사양이면 5s로 자동 완화
- **검증**: 저사양(Celeron·구형 i3) 노트북에서 실측 리허설
- **잔여 리스크**: 응시자 노트북 상태 (백그라운드 앱 · 발열)

### 3.8 Supabase Realtime 채널 요금

- **원인**: 300 응시자 outbound 채널 유지 시 요금 급증
- **대응**:
  - 응시자는 outbound POST만 (Realtime channel 미사용)
  - 관리자 대시보드는 5초 폴링 (Realtime channel 소수)
  - 개별 상세 진입 시에만 Realtime channel 개별 구독
- **검증**: 부하 시나리오에서 Realtime 채널 총 수 < 30 확인

- 답안 유실 0 · p95 < 목표 · 시험 종료 시각 최종 제출 폭주 안전
- 사전 100명 리허설 → 문제 발견 → 150명 리허설 → 시험 배포

### 4.5 프로덕션 실전 시험 첫 회

- 첫 회는 응시자 100~150명으로 시작 (승우님·다에이지 판단)
- 무사 통과 → 그다음 시험부터 300 명 배포

## 8. 결론

- **300명 동시 응시는 기술적으로 가능**하다 (Supabase Pro + PgBouncer + Daily Prime + R2 + 클라 배치 전략)
- **최대 리스크는 Daily.co 비용** (월 100~150만원 예상) — 세일즈 견적으로 첫 검증
- **다음 리스크는 실기기 리허설** — 회사에서 노트북 100~150대 확보 필요
- 브라우저 부하는 대기실 벤치·최소 사양·예외 승인으로 관리
- M1 진입 전 미결정 5개만 확정하면 나머지 스택은 명확
